# 前后端联调参数治理规则

> 适用对象：前端开发、后端开发、联调测试
>
> 目标：在联调阶段明确“哪些参数可以由前端传入，哪些必须以后端认证上下文为准”，避免身份、权限、租户、审计字段被前端错误控制。

---

## 1. 核心判断标准

联调时先把参数分成两类：

- 资源定位参数：表示“我要操作哪条业务数据”
- 上下文控制参数：表示“我是谁、我有什么权限、我属于哪个数据域、这次操作由谁产生”

规则非常简单：

- 前端可以传资源定位参数
- 前端不应该决定上下文控制参数
- 后端必须基于认证上下文自行判断身份、角色、数据范围和审计信息

可以压缩成一句话：

> 前端负责指定目标资源，后端负责确认操作者身份、权限、作用域和审计归属。

---

## 2. 五条联调原则

### 原则 1：前端可传业务对象 ID，不可决定身份和权限

前端可以传：

- `activityId`
- `membershipId`
- `organizationId`
- `volunteerId`
- `orderId`
- `employeeId`

这些字段表达的是“要操作哪条记录”。

前端不应决定：

- `userId`
- `role`
- `permission`
- `operatorId`

这些字段表达的是“是谁在操作、是否有权限操作”，必须由后端判断。

### 原则 2：用户身份、租户、角色、组织信息以后端认证上下文为准

以下字段默认不信前端：

- `userId`
- `accountId`
- `role`
- `permission`
- `tenantId`
- `orgId`
- `hospitalId`

这些信息如果参与数据过滤、权限判断、租户隔离，必须由后端从登录态、令牌、会话或网关透传上下文中获取。

### 原则 3：前端传来的资源 ID，后端必须做存在性和归属校验

只要前端传了资源 ID，后端至少要做两件事：

1. 校验资源是否存在
2. 校验当前登录用户是否有权访问该资源，且资源是否属于当前作用域

举例：

- 前端传 `activityId`，后端要校验该活动是否存在
- 前端传 `organizationId`，后端要校验当前账号是否能访问该组织
- 前端传 `volunteerId`，后端要校验当前账号是否可查看该志愿者信息

### 原则 4：审计字段由后端生成，前端不参与控制

以下字段默认应由后端生成或覆盖：

- `createdBy`
- `updatedBy`
- `operatorId`
- `createdAt`
- `updatedAt`

前端可以展示这些字段，但不应该决定这些字段的最终值。

### 原则 5：查询条件中涉及数据隔离的字段，后端强制追加，不依赖前端传参

如果一个查询接口涉及租户隔离、组织隔离、角色范围或“只能看我自己的数据”，则后端必须在查询条件中主动追加这些限制，不依赖前端自己传：

- 当前租户范围
- 当前组织范围
- 当前用户可访问范围
- 当前账号所属业务实体

即使前端传了相关字段，后端也只能把它当“候选过滤条件”，不能把它当最终隔离条件。

---

## 3. 联调时重点检查的四类字段

建议联调时优先把参数按下面四类过一遍。

### 3.1 身份类

典型字段：

- `userId`
- `accountId`
- `role`
- `permission`

联调规则：

- 默认不信前端
- 不把它们作为最终权限依据
- 后端以认证上下文为准

### 3.2 租户类

典型字段：

- `tenantId`
- `orgId`
- `hospitalId`

联调规则：

- 默认不信前端
- 后端必须自行注入或覆盖
- 查询时必须强制追加隔离条件

### 3.3 审计类

典型字段：

- `createdBy`
- `updatedBy`
- `operatorId`

联调规则：

- 默认不信前端
- 后端统一生成
- 审计日志和数据库落库值以后端为准

### 3.4 资源类

典型字段：

- `employeeId`
- `patientId`
- `orderId`
- `activityId`
- `membershipId`
- `volunteerId`
- `organizationId`

联调规则：

- 可以接收前端传入
- 必须校验存在性
- 必须校验权限
- 必须校验归属

前三类默认不信前端，第四类可以接收，但必须校验权限和归属。

## 4. 当前项目的落地解释

本项目后端模型已经区分了：

- 账户表
- 志愿者表
- 组织者表

合理链路应该是：

1. 后端先从认证上下文获取 `accountId`
2. 再根据 `accountId` 判断账户是否存在
3. 再根据账户关联去查询志愿者或组织实体
4. 最终决定当前请求对应的业务主体和可访问范围

所以前端需要遵守的边界是：

- `accountId` 属于身份上下文，不应用来冒充业务实体 ID
- `volunteerId`、`organizationId` 属于业务资源 ID，只能在确实需要定位资源时使用
- “当前我是哪个志愿者/组织者” 应以后端解析结果为准

---

## 5. 当前项目中的字段分类建议

### 5.1 应以后端上下文为准的字段

- `accountId`
- `userId`
- `role`
- `permission`
- `operatorId`
- `createdBy`
- `updatedBy`

这些字段不应由前端决定最终值。

### 5.2 可由前端传入的业务资源字段

- `activityId`
- `organizationId`
- `membershipId`
- `volunteerId`
- `auditId`
- `session_id`

这些字段可以由前端从列表、详情、搜索结果或页面当前记录中获取后传给后端。

前提是：

- 它们来自后端已经返回的数据
- 不是前端自己凭空推导身份
- 后端仍然会做权限和归属校验

---

## 6. 接口设计建议

### 6.1 更适合纯上下文的接口

如果接口语义是“当前登录用户自己的数据”，优先做成纯上下文接口。

例如：

- 我的首页
- 我的资料
- 我的组织关系
- 我的报名记录
- 我的实名提交
- 我的签到/签退

这类接口更适合：

- 不传 `userId`
- 不传 `accountId`
- 尽量不传 `volunteerId`

而是由后端从上下文解析当前业务主体。

### 6.2 更适合显式传资源 ID 的接口

如果接口语义是“操作某条业务记录”，则应显式传资源 ID。

例如：

- 查看活动详情：传 `activityId`
- 更新活动：传 `activityId`
- 退出组织关系：传 `membershipId`
- 查看组织详情：传 `organizationId`
- 查看志愿者详情：传 `volunteerId`

这类接口允许前端传目标资源，但后端必须补做校验。

---

## 7. 联调时的快速检查清单

每看一个接口，都建议按下面四步快速过一遍：

1. 这个参数是身份/权限/隔离字段，还是资源定位字段？
2. 如果是身份类字段，能否改为由后端上下文获取？
3. 如果是资源类字段，它是否来自后端已返回的数据？
4. 后端是否对这个资源做了存在性、权限、归属校验？

如果一个接口同时出现了“身份类字段”和“资源类字段”，优先警惕：

- 身份字段是否被前端越权控制
- 资源字段是否缺少归属校验

---

## 8. 反例与正例

### 8.1 反例

前端直接传：

```json
{
  "userId": 12,
  "role": "admin",
  "operatorId": 12,
  "organizationId": 5
}
```

问题：

- 前端在决定操作者身份
- 前端在决定角色
- 前端在决定审计归属

这三项都不应被信任。

### 8.2 正例

前端只传目标资源：

```json
{
  "organizationId": 5
}
```

后端再自行完成：

- 从上下文取当前登录账户
- 判断该账户角色
- 判断是否能访问组织 `5`
- 记录本次操作的审计信息

---

## 9. 本文档的使用方式

联调建议同时配合两份文档使用：

- 接口清单文档：看接口路径、请求字段、返回结构
- 本规则文档：判断字段应不应该由前端传、后端该不该信

接口文档回答的是：

- 这个接口怎么调
- 传什么
- 返回什么

本规则文档回答的是：

- 这个字段该不该由前端控制
- 这个字段后端是否必须覆盖或校验
- 这个接口应该更偏向“上下文型”还是“资源型”

---

## 10. 最终结论

联调时不要把“能不能传 ID”当成唯一问题。

真正要区分的是：

- 传的是资源 ID，还是身份/权限/隔离 ID

结论如下：

- 资源 ID 可以传
- 身份、权限、租户、审计字段默认不信前端
- 后端必须用认证上下文决定操作者身份和数据范围
- 后端必须对前端传入的资源 ID 做存在性和归属校验

这样前后端职责才会清晰，联调也更稳定。
